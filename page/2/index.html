一些知识点

idea 下部署项目的访问路径指定

tomcat 启动后，会自动读取 webapp 路径下的所有文件夹，每一个文件夹它都会视作一个项目，且从 webapps 开始的文件夹路径就是这个项目的访问路径。

如 webapp 下有一个叫bitTemp的文件夹，则 tomcat 启动后，就可以通过http://localhost:8080/bitTemp这个路径访问这个 bitTemp 项目，同理，如果这个文件夹多套了一层，实际路径变成${TOMCAT_HOME}/webapps/biteTemp/bitTemp2，则需要通过http://localhost:8080/bitTemp/bitTemp来访问原先的页面。即只看文件夹的名称与路径。

idea 部署 tomcat 项目，实际上就是把生成的 target 包复制到 tomcat 的 webapp 下，而决定复制到 webapps 下的文件名的就是Application context这个参数。

Deployment 里，war包与war exploded包部署的区别

  • war模式可看做发布模式，将成果先打成 war 包，再发布到 tomcat 里。
  • war exploded模式就是让项目在编译完成后，使 tomcat 直接来 target 路径读取整个项目文件，无需 war 包，也无需移动文件。
    • 因此war exploded方式支持热部署，开发时会比较方便。

实际用下来不使用热部署的话，两种方式区别其实不大

要支持热部署，tomcat 本身还需要做一定设置。

windows 下 idea 控制台的 tomcat 日志中中文乱码

tomcat 日志的编码问题，调整一下 tomcat 日志的配置文件即可。
日志配置文件路径为${TOMCAT_HOME}/conf/logging.properties。

windows 下使用

windows 下直接使用

  1. cd 进入 redis 的目录
  2. 打开 cmd 窗口
  3. 执行redis-server.exe redis.windows.conf

这样执行后， redis 服务将在这个 cmd 页面上开启。使用同目录下的redis.windows.conf作为配置文件。
当这个 cmd 窗口关闭后，redis 服务就会停止。
要让 redis 服务常驻，需要在 windows 下注册服务。

windows 下注册服务

以下命令都需要在 redis 的目录下执行，否则需要指定各文件的绝对路径。

  1. 注册服务:redis-server.exe --service-install redis.windows.conf
  2. 卸载服务:redis-server --service-uninstall
  3. 注册服务后，开启 redis 服务:redis-server --service-start
  4. 注册服务后，停止 redis 服务:redis-server --service-stop

常用的 redis.conf 中的配置

配置项 值 描述
daemonize yes/no 是否让 redis 在后台运行的，默认为 no ，如果需要在后台运行，把该项的值更改为 yes。
protected-mode yes/no 是否开启保护模式，即禁止除本机以外的 IP 通过网络访问该 redis 节点。如需正常使用，需要置为 no，注意，若要通过网络访问 redis 节点，还需要同步修改 bind 项的配置
bind IP值(127.0.0.1) redis 仅接受来自指定 IP 的请求，不设置则默认为0.0.0.0，即接受所有请求。多个 IP 间需要通过空格做分割
port 端口号 监听端口，默认 6379

主从复制模式

集群模式的一种，实现数据的多节点备份，支持对读操作的负载均衡。
缺陷在于：故障恢复无法自动（需手动操作），写操作无法负载均衡，存储能力受单节点限制。

所有节点都支持读，但只有主节点支持写，因此写依旧是单节点的。

主从复制模式部署

每个节点分开部署，可以在一个服务器上部署多个 redis 来简单构建集群。
每个 redis 节点的部署都可以按照上面的步骤来（同服务器下的话直接复制 bin 目录也行，redis.conf文件需要改端口号)

相比单节点部署，主从复制模式要在redis.conf配置文件里改一些内容

  • 主节点: 什么都不需要做
  • 从节点: 修改redis.conf，找到里面的REPLICATION模块，添加一项slaveof <MASTER IP> <MASTER PORT>，修改完成后，重启这个 redis 节点
    • 也可以不修改配置文件，在从节点启动的情况下，用redis-cli连接这个节点，执行一下slaveof <MASTER IP> <MASTER PORT>，也是一样的效果

从节点修改完后，用redis-cli连接一下主节点，执行一下info Replication，可以查看集群的状态（以及连接的节点是主节点还是从节点）

主从哨兵模式的部署，有配置文件详解:https://blog.csdn.net/m0_38110132/article/details/85168960

哨兵模式

集群模式的一种，基于主从模式，实现了自动化的故障恢复。
缺陷在于：写操作无法负载均衡，每个节点都存放了全量数据，存储能力受单节点限制。

哨兵模式的一些注意点

  1. sentinel monitor mymaster 127.0.0.1 6379 2这一行中填写的主节点 IP 与 PORT，后续会被用于发送给连接哨兵的客户端，用于客户端连接 redis 节点。因此直接写 127.0.0.1 会导致远程连接的 redis 客户端发生问题(直接读取到 127.0.0.1)
  2. 哨兵模式下，JAVA 的 redis 客户端连接 redis 集群时，填写的不能是 redis 节点的 IP，必须是哨兵集群的 IP 与 PORT。
    • 哨兵模式下也不是不能直接填写 redis 节点的 IP，但这样就只能当作单节点来连接（即普通的主从模式），不再拥有哨兵模式的特性。
    • 虽然填写的是哨兵的 IP，但依旧实际连接时依旧需要直连 redis 的主节点。因此当需要开通接口策略的时候，还是得哨兵和主从节点的端口都给开了

本质就是外置哨兵进程来监控 redis 集群进程的状态，外加故障转移机制。

cluster 集群模式

相比哨兵模式更近一步（并不完全基于哨兵模式），多个主节点分布式存放数据，且也拥有自动化的故障恢复功能。
cluster 集群模式里不再有专门的哨兵，每个主节点兼职了哨兵的功能。
因为每个主节点存放不同的数据，因此一个节点挂掉后就不能简单放弃这个节点，所以每个主节点都需要至少一个专属于自己的 slave 节点，用于故障转移。
因此 cluster 集群模式至少需要三主三从，即至少六个节点。

cluster 集群模式部署

cluster 集群模式根据 redis 版本不同，存在两种部署方式。
5.0.0 以前的版本需要用到集群工具(ruby 编写，位于 redis 源码包，使用前需要先安装 ruby)，5.0.0 以后不需要另外安装 ruby，可以直接调用命令构建集群。
文档以 5.0.14 版本为例。

集群模式的坑

网络策略严格的情况下会遇到的问题

在腾讯云的机器上执行这样的命令。理论上可以建立集群，但实际情况卡在了Waiting for the cluster to join这一步（持续等待）。
原因是：redis 集群不仅需要开通redis客户端连接的端口，而且需要开通集群的总线端口。
总线端口为：redis 节点的端口 + 10000，且几个节点就需要开几个总线端口。
上述的命令有 6 个端口，因此还需要在防火墙上额外开 6 个端口。

这个问题目前仅发生在 redis 集群指定为外网 IP 的时候。当 IP 指定为 127.0.0.1 时，没有这个问题（但指定为 localhost 也没法从外部访问了，没啥意义）。

启动报错

当前用户允许拥有的虚拟内存的大小不足。

需要调整vm.max_map_count的值。该调整需要 root 权限。

max_map_count文件包含 限制一个进程可以拥有的 VMA ( 虚拟内存区域 ) 的数量 。
虚拟内存区域是一个连续的虚拟地址空间区域。在进程的生命周期中，每当程序尝试在内存中映射文件，链接到共享内存段，或者分配堆空间的时候，这些区域将被创建。
调优这个值将限制进程可拥有 VMA 的数量。 限制一个进程拥有 VMA 的总数可能导致应用程序出错 ， 因为当进程达到了 VMA 上线但又只能释放少量的内存给其他的内核进程使用时，操作系统会抛出内存不足的错误 。如果你的操作系统在 NORMAL 区域仅占用少量的内存，那么调低这个值可以帮助释放内存给内核用。

初始堆大小不等于最大堆大小。

本质是自己改配置改出来的问题，config/jvm.options中的-Xms与-Xmx的值不一致。
这两个值的默认值同为 4G, 内存足够全用默认值的话其实不会出错。

缺少默认配置discovery settings

看提示可知：缺少默认配置，至少需要配置discovery.seed_hosts，discovery.seed_providers，cluster.initial_master_nodes中的一个参数，但实际修复过程发现远不止这一个参数。

  • discovery.seed_hosts: 集群主机列表,单机模式的话就在这里填一个节点即可
  • discovery.seed_providers: 基于配置文件配置集群主机列表
  • cluster.initial_master_nodes: 启动时初始化的参与选主的 node，集群模式需要。与配置文件中的node这一项关联。

Nginx 的安装与部署

参考链接: https://www.cnblogs.com/lywJ/p/10710361.html

简介

Nginx 是一款高性能的 http 服务器/反向代理服务器及电子邮件（IMAP/POP3）代理服务器。
由俄罗斯的程序设计师 Igor Sysoev 所开发，官方测试 nginx 能够支支撑 5 万并发链接，并且 cpu、内存等资源消耗却非常低，运行非常稳定。

作用

  • http 服务器：Nginx 是一个 http 服务可以独立提供 http 服务。可以做网页静态服务器。
  • 虚拟主机：可以实现在一台服务器虚拟出多个网站。例如个人网站使用的虚拟主机。
  • 反向代理，负载均衡：当网站的访问量达到一定程度后，单台服务器不能满足用户的请求时，
    • 需要用多台服务器集群可以使用 nginx 做反向代理。并且多台服务器可以平均分担负载，不会因为某台服务器负载高宕机而某台服务器闲置的情况。
  • 内网转发请求，达成外网直接访问内网应用的目的。

安装与部署

下载

官方网址：http://nginx.org/en/download.html

Nginx官网提供了三个类型的版本

  • Mainline version：Mainline 是 Nginx 目前主力在做的版本，可以说是开发版
  • Stable version：最新稳定版，生产环境上建议使用的版本
  • Legacy versions：遗留的老版本的稳定版

注意：下载的都是源码包，需要自行编译。

好像确实有不通过源码包，直接安装的方法，之后研究下。

上传并解压

测试版本用的是稳定版本 nginx-1.20.2，上传压缩包至服务器，路径随意（只是源码包，编译出来后的才是部署本体）。

直接解压。

编译并部署

先创建一下存放临时文件的文件夹，下一个步骤会用到。

使用.\configure命令创建一个MakeFile文件。注意，执行.\configure命令时必须进入解压出来的文件夹里去。

\ 其实只是续写标识符，表明这条命令没写完，下一行接着写，所以本质上上面这一块只是一条命令。
--prefix=/usr/local/nginx这一行指定了编译生成的 nginx 部署成果的路径，有必要的话可以改。
后续跟着的所有参数其实都是需要添加的 nginx 的功能（类似 plugin）。
最后一行的--with-stream非常必要，用于转发各种 TCP，UDP 请求的核心，内外网映射数据库端口就需要这个。

执行完上述命令后，会在当前目录下看到一个Makefile文件，具体作用不太清楚，反正跟编译有关。

执行make命令，执行编译，成功后再执行make install安装部署 nginx 成果。

仅仅执行一个 make 命令，什么文件也不会生成。

make install执行完成后，nginx 即部署完成，可以进入安装目录启动 nginx。

上下文路径

docker 的运行模式为 C/S，本机是 C，docker 引擎是 S，实际的镜像构建由 docker 引擎完成，因此引擎无法直接使用本机的文件。因此需要把所需的文件发送给引擎。
这个上下文路径就是指定了需要发送给 docker 引擎的文件的路径。
使用docker build命令后，这个路径下的所有文件都会发送给 docker 引擎，供 docker 引擎构建 image 使用。
注意：构建 image 时，docker 引擎只会使用到位于上下文路径内的文件。

所以指定的路径下不应有多余的文件，虽然不影响最后生成的 image，但会拖慢构建 image 的速度（打包发送的时间变长）。

Dockerfile 常用指令

FROM

指定基础镜像，表示待自定义的镜像从这个基础镜像上修改而来。
一般是 Dockerfile 的第一行。

RUN

用于执行后面紧跟着的命令，有两种格式。
shell 格式:

exec 格式:

CMD

运行程序，但与 RUN 指令的运行时间点不同。

  • RUN 会在 docker build 阶段执行
  • CMD 会在 docker run 时运行

作用：为启动的容器指定默认要运行的程序，程序运行结束，容器也就结束。CMD 指令指定的程序可被 docker run 命令行参数中指定要运行的程序所覆盖。
注意：如果 Dockerfile 中如果存在多个 CMD 指令，仅最后一个生效。

ENTRYPOINT

指定 docker run 时需默认运行的程序与默认参数。若不指定，则默认为/bin/sh

这个参数一般与 CMD 指令结合使用，ENTRYPOINT 中指定程序与固定参数，CMD 中指定变参。
如：启动 tomcat

LABEL

键值对，用于给镜像添加一些元数据，没什么大用的样子。可以用来标识镜像作者。

ENV

环境变量，定义之后，该变量将直接写入生成的镜像内（Dockerfile 内也有效，即 docker build 的过程中也能用）。

ARG

构建参数，作用与 ENV 一致，但 ARG 中定义的环境变量仅在 Dockerfile 内生效，即只在 docker build 过程中有效。
最后生成的 image 中不会有对应的环境变量。

默认值可以设定也可以不设，不设定时，需要在docker build时使用参数--build-arg <参数名>=<值>来补充入参。注意：必须先在 Dockerfile 里定义了 ARG 参数，--build-arg才能生效

WORKDIR

指定工作目录，由 WORKDIR 指定的工作目录将存在于构建镜像的每一层（WORKDIR 只能指定工作目录，不能自动创建，因此必须提前建好）。

相关内容：镜像与层

Docker 的 image 实际是由一系列的层来构成的，Dockerfile 里的每一条命令（包括 FROM，ENV 等）都会生成新的一层。
所谓自定义 image，其实也就是在别人的 image 上多套几层。
容器由 image 生成，容器实际上也是在 image 上多套了一层可读写的层（image 里的层对容器来说都是只读的），这样就保证了 image 的一致。

也是因为层的原因，比较推荐能写成一行的命令就写成一行，这样层数少一点，可以减小镜像的大小

COPY

复制指令，从上下文目录中复制文件或者目录到容器里指定路径，指定的目标路径不存在时，会自动创建。

允许通过--chown=<user>:<group>可选参数，改变复制到容器内文件的拥有者和属组。
<源路径>允许是文件或者目录，支持通配符，但通配符的格式需满足 Go 的filepath.Match规则（没整明白这是什么规则）

EXPOSE

声明端口。运行时若使用随机端口映射(docker run -P)，则只会在 EXPOSE 中声明的端口列表内随机。

ADD

作用与 COPY 一致，只是针对压缩文件，使用 ADD 命令，builder 会直接解压，且若解压失败，镜像构建就会直接失败。
所以更推荐直接用 COPY。
