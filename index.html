Typora免费版本安装流程

几个背景

最新的免费版本0.11.18(安装版本是0.11.17，反正差别应该不大)，安装包比较坑爹，只能 install 且不能自己选择安装路径，只能放进 C 盘。

官网已无法下载到 Typora 的免费版本，安装包只能在各个网盘、网站里翻找。

安装流程

安装包下载完成后，直接点击打开，即可安装，安装时会提示是给全部用户安装还是仅给当前用户，可以只选给当前用户安装（应该区别不大）。

然后自动读条安装，安装完成后会自动打开，此时将会发生报错。

点击确定后自动跳转到 Typora 的官网，且 Typora 不会正常被打开。

后续所有打开 Typora 的操作都会发生这样的报错并失败（就是官方不让用旧版本了）。

解除测试版本启动时的版本检查

其实就是修改注册表权限，禁止 Typora 访问注册表。

  1. windows 下，windows 键 + R，输入 regedit，打开注册表。

  2. 在 HKEY_CURRENT_USER\SOFTWARE\ 下找到 Typora 目录。

  1. 右键注册表这边的 Typora 目录，打开 Typora 权限设置。将权限修改为拒绝（可以把所有用户访问的权限都设置成拒绝)。

设置完成后，Typora 就可以正常启动了。

针对普通用户

  1. root 用户下，打开/etc/passwd文件
    • vim /etc/passwd
  2. 找到需要对应的用户(一般从下往上找，或者直接搜索也行)，如 kafka, 可以在文件内看到这样的内容
    • kafka:x:1002:1002::/home/kafka:/bin/bash
  3. 将这一行改成如下的内容，然后保存
    • kafka:x:1002:1002::/home/kafka:/usr/bin/nologin
  4. root 用户下，修改/etc/shells文件，末尾新增一行内容并保存
    • /usr/bin/nologin
  5. 无需重启服务器或服务，该用户即被禁止远程登陆

注意事项

  1. 简单禁用指定用户 ssh 方式的登陆，只需要执行上述的 1, 2, 3 这三个步骤（即不修改/etc/shells文件），但不执行第 4 步，将无法禁止用户通过 FTP 方式远程登陆。

针对 root 用户

禁止 root 用户远程登录，需要修改/etc/ssh/sshd_config文件，找到这一块内容。

将其中的PermitRootLogin yes这一行，由yes改成no。
改完后保存，然后重启 sshd 服务。

Hexo解决图片无法正常显示的问题

Hexo 支持 markdown 格式的文件做页面，但通过 hexo 命令生成 html 时，会由于文档中图片路径不对，生成的网页无法正常显示图片。

Hexo 本地部署的方式，其实就是把 ${HEXO_HOME}/source/_post 文件夹下的 markdown 文件转成 html，统一放到 public 文件夹下。而不会复制其中的图片文件，这才导致了错位。

解决这个问题，需要下载一个插件hexo-renderer-marked。

下载并启用 hexo-renderer-marked 插件

先下载安装插件。

修改 Hexo 的_config.yml文件，修改其中的 writting 部分内容。

hexo-renderer-marked 的官方文档（英文）：https://www.npmjs.com/package/hexo-renderer-marked

将post_asset_folder置为 true 后，每次执行hexo g时，hexo 才会将资源文件夹中的内容复制到 public 里去，且与生成的 html 放在同一个路径下。注意：即便post_asset_folder置为 true，hexo 也只会读取与文章同名的文件夹，将其视为资源文件，若资源文件夹与 md 文件不同名，则不会复制。

文章中图片路径写法的注意事项

上述方式可以解决图片完全不会被打包发布的问题，但可以看到图片与文章的绝对路径与相互之间的相对路径都是改变了的（执行hexo g后，生成的所有静态文档都是放到 public 文件夹内的，且资源文件夹内的资源会与文章放在一个文件夹下，同一篇文章只会有一个文件夹，里面塞了所有的图片与 唯一的html）。

然而 html 文件是通过 md 文件生成的，生成后并不会主动改变图片的指向路径，所以：采用绝对路径的，可以在本地执行hexo s时继续看到文件（但是没用，发布到 github page 上就看不到了，因为对应路径不存在了），因此必须采用相对路径，且需要执行hexo g后依然能不会错的相对路径（即必须在同一个文件夹下）。

但这样又会有一个问题：如果在 md 文件里就写成最终生成时应该有的图片相对路径，写文章直接预览 md 文件时，图片又会看不到（因此此时图片文件还在同名文件夹里）。

因此需要一个兼容方案，既可以本地 md 正常预览图片，又可以在部署后正常查看 html 文章。

typora 解决本地远程图片兼容读取的方案

目前只能通过 typora 来解决这个问题，因为 vscode 没有对应的功能。

具体操作就是通过 typora 打开 md 文件，然后设置文章的图片根目录。将其修改为资源文件夹。

（格式 -> 图样 -> 设置图片根目录 -> 选中同名资源文件夹）

完成后，typora 会在文章抬头出现这样一行内容：typora-root-url: Hexo解决图片无法正常显示的问题

这个的逻辑主要就是将指定的文件夹视作图片的根目录（原本肯定是 md 文件所在的路径），等于是修改了 md 文件渲染的规则，目前只有 typora 能简单做到对单篇文章的快速修改图片根目录。

所以此后，图片文件与 md 文件可以视作放在一个文件夹里，本地与远程图片读取可兼容。

更简便设置文章图片根目录的方法

可以利用 hexo 的模板文件，在模板的开头直接加上这么一行:typora-root-url:{{ title }}

这样以后通过hexo new 文章的方式，生成出来的 md 文件将自动设置好对应的图片根路径。

基本介绍

Hexo 是一个快速、简洁且高效的博客框架。Hexo 使用 Markdown（或其他渲染引擎）解析文章，在几秒内，即可利用靓丽的主题生成静态网页。

  • Git

可以将 Hexo 的对应路径写入环境变量 PATH 里，之后就可以直接使用 hexo <command>

指定一个文件夹，执行下面的命令， Hexo 将在指定的文件夹内新建所需的文件。

部分文件夹与文件用途的简介

  • _config.yml: 网站的配置信息。大部分配置参数都在此处设置。
  • scaffolds: 模板文件夹，通过hexo new page xxx方式新建文章时，Hexo 会根据 scaffolds 里的内容来生成默认内容(时间戳，用户名，标签、分类、默认字体等)。
  • source: 资源文件夹，所有的文章的源文件都在这里。除_posts文件夹之外，开头命名为_(下划线)的文件 / 文件夹和隐藏的文件将会被忽略。Markdown 和 HTML 文件会被解析并放到 public 文件夹，而其他文件会被拷贝过去。
  • themes: 主题文件夹，Hexo 会根据主题来生成静态页面，可以理解成不同的 css 包。

发布文章

新建文章

新建文章可以直接在 source 文件夹内新建文件（建子文件夹再建文件也可以）。或者通过命令新建，通过命令的方式可以使用模板。

以hexo new photo "My Gallery"这一命令为例，Hexo 会在scaffold文件夹内寻找photo.md，并根据其中的内容建立文章。

大部分情况下，其实直接手动建文件夹建文件就够用了，从 index.html 里复制一下就行。

Hexo 远程部署

主要是把 Hexo 网站部署到 GitHub 上去，使用 GitHub Page 的功能直接远程访问博客。

GitHub 新建仓库

使用 GitHub Page，必须要用指定的仓库(repository)，即: 用户名.github.io。
访问权限需要是 public。

对应的仓库建立完成后，应该是可以直接访问 GitHub Page 的默认网站的，网站路径:用户名.github.io 。

修改 Hexo 配置，设置上传到 GitHub 的路径

打开 Hexo 的配置文件_config.yml,找到deploy这一块，配置上新建的 GitHub 仓库的 url。

deploy:
type: git
repo: git@github.com:move-mountains/move-mountains.github.io.git
branch: master

具体上传到哪个分支、 GitHub Page 怎么选择分支还没研究过

使用 Hexo 命令上传

主要用法是一键部署。需要安装插件hexo-deploy-git。

在本地 Hexo 博客的根目录下，打开 cmd，执行命令，上传。

上述命令也可以简写成hexo d。该命令实际集成了一部分的 git 操作，Hexo 会把 public 目录中的文件和目录推送到_config.yml里配置的远程仓库和指定分支，并完全覆盖该分支下的所有内容。

注意： _config.yml里可以配置多条 git 的 deploy 路径，但在用hexo deploy方式上传时，对应的分支必须已经存在，否则会报错(这样的 deploy 方式似乎不会主动新建分支)。

redis 部署

redis 支持单机模式、主从复制模式、哨兵模式、cluster集群模式这几种部署方式。
单机模式最简单，但是不具备高可用。

配置 redis 可远程访问

在redis.conf的配置文件里修改# bind 127.0.0.1这一行，改为0.0.0.0，表示支持任意 IP 的访问。

远程访问 redis

需要使用redis-cli文件，windows 下是redis-cli.exe，linux 下是redis-cli。

无密码可去掉最后的-a password

注意：不允许远程访问 redis 时，使用 redis-cli 尝试连接也不会报错，但会在执行操作的时候报错

单机模式

windows 下部署

下载压缩包后解压,解压完即可使用。

windows 下使用

windows 下直接使用

  1. cd 进入 redis 的目录
  2. 打开 cmd 窗口
  3. 执行redis-server.exe redis.windows.conf

这样执行后， redis 服务将在这个 cmd 页面上开启。使用同目录下的redis.windows.conf作为配置文件。
当这个 cmd 窗口关闭后，redis 服务就会停止。
要让 redis 服务常驻，需要在 windows 下注册服务。

windows 下注册服务

以下命令都需要在 redis 的目录下执行，否则需要指定各文件的绝对路径。

  1. 注册服务:redis-server.exe --service-install redis.windows.conf
  2. 卸载服务:redis-server --service-uninstall
  3. 注册服务后，开启 redis 服务:redis-server --service-start
  4. 注册服务后，停止 redis 服务:redis-server --service-stop

编译通过后，再执行正式的构建命令

PREFIX=/opt/redis/redis指定了安装的可执行文件的路径。成功后可以直接到这个目录下看到一个bin文件夹。理论上应该只有这么一个文件夹。
bin 文件夹内的内容如下。

各文件的作用如下:

  • redis-server：redis 服务器的 daemon 启动程序。
  • redis-cli：Redis命令操作工具。当然，也可以 telnet 根据其纯文本协助来操作。
  • redis-benchmark：Redis性能测试工具，测试 Redis 在你的系统及你的配置下的读写性能。
  • redis-check-aof：更新日志检查。
  • redis-check-dump：用于本地数据库检查。

想自定义配置来运行 redis,需要从 执行过make后的 redis 安装包中复制redis.conf文件过来（执行过make后就会自动生成，所有模板配置文件都能在这个目录下找到，包括哨兵的sentinel.conf）
更完整的 redis 配置，可以自己在 bin 的同级目录下建 conf, log, run, data 文件夹。分别用于存放配置文件，日志文件，运行时的 pid 文件，快照与 AOF 文件，然后分别在 redis.conf 文件夹内指定。

常用的 redis.conf 中的配置

配置项 值 描述
daemonize yes/no 是否让 redis 在后台运行的，默认为 no ，如果需要在后台运行，把该项的值更改为 yes。
protected-mode yes/no 是否开启保护模式，即禁止除本机以外的 IP 通过网络访问该 redis 节点。如需正常使用，需要置为 no，注意，若要通过网络访问 redis 节点，还需要同步修改 bind 项的配置
bind IP值(127.0.0.1) redis 仅接受来自指定 IP 的请求，不设置则默认为0.0.0.0，即接受所有请求。多个 IP 间需要通过空格做分割
port 端口号 监听端口，默认 6379

更多 redis 配置项与用途: https://blog.csdn.net/weixin_40482816/article/details/106262586

后台启动 redis 服务

主从复制模式

集群模式的一种，实现数据的多节点备份，支持对读操作的负载均衡。
缺陷在于：故障恢复无法自动（需手动操作），写操作无法负载均衡，存储能力受单节点限制。

所有节点都支持读，但只有主节点支持写，因此写依旧是单节点的。

主从复制模式部署

每个节点分开部署，可以在一个服务器上部署多个 redis 来简单构建集群。
每个 redis 节点的部署都可以按照上面的步骤来（同服务器下的话直接复制 bin 目录也行，redis.conf文件需要改端口号)

相比单节点部署，主从复制模式要在redis.conf配置文件里改一些内容

  • 主节点: 什么都不需要做
  • 从节点: 修改redis.conf，找到里面的REPLICATION模块，添加一项slaveof <MASTER IP> <MASTER PORT>，修改完成后，重启这个 redis 节点
    • 也可以不修改配置文件，在从节点启动的情况下，用redis-cli连接这个节点，执行一下slaveof <MASTER IP> <MASTER PORT>，也是一样的效果

从节点修改完后，用redis-cli连接一下主节点，执行一下info Replication，可以查看集群的状态（以及连接的节点是主节点还是从节点）

哨兵模式

集群模式的一种，基于主从模式，实现了自动化的故障恢复。
缺陷在于：写操作无法负载均衡，每个节点都存放了全量数据，存储能力受单节点限制。

哨兵模式的一些注意点

  1. sentinel monitor mymaster 127.0.0.1 6379 2这一行中填写的主节点 IP 与 PORT，后续会被用于发送给连接哨兵的客户端，用于客户端连接 redis 节点。因此直接写 127.0.0.1 会导致远程连接的 redis 客户端发生问题(直接读取到 127.0.0.1)
  2. 哨兵模式下，JAVA 的 redis 客户端连接 redis 集群时，填写的不能是 redis 节点的 IP，必须是哨兵集群的 IP 与 PORT。
    • 哨兵模式下也不是不能直接填写 redis 节点的 IP，但这样就只能当作单节点来连接（即普通的主从模式），不再拥有哨兵模式的特性。
    • 虽然填写的是哨兵的 IP，但依旧实际连接时依旧需要直连 redis 的主节点。因此当需要开通接口策略的时候，还是得哨兵和主从节点的端口都给开了

本质就是外置哨兵进程来监控 redis 集群进程的状态，外加故障转移机制。

cluster 集群模式

相比哨兵模式更近一步（并不完全基于哨兵模式），多个主节点分布式存放数据，且也拥有自动化的故障恢复功能。
cluster 集群模式里不再有专门的哨兵，每个主节点兼职了哨兵的功能。
因为每个主节点存放不同的数据，因此一个节点挂掉后就不能简单放弃这个节点，所以每个主节点都需要至少一个专属于自己的 slave 节点，用于故障转移。
因此 cluster 集群模式至少需要三主三从，即至少六个节点。

cluster 集群模式部署

cluster 集群模式根据 redis 版本不同，存在两种部署方式。
5.0.0 以前的版本需要用到集群工具(ruby 编写，位于 redis 源码包，使用前需要先安装 ruby)，5.0.0 以后不需要另外安装 ruby，可以直接调用命令构建集群。

集群模式的坑

网络策略严格的情况下会遇到的问题

在腾讯云的机器上执行这样的命令。理论上可以建立集群，但实际情况卡在了Waiting for the cluster to join这一步（持续等待）。
原因是：redis 集群不仅需要开通redis客户端连接的端口，而且需要开通集群的总线端口。
总线端口为：redis 节点的端口 + 10000，且几个节点就需要开几个总线端口。
上述的命令有 6 个端口，因此还需要在防火墙上额外开 6 个端口。

这个问题目前仅发生在 redis 集群指定为外网 IP 的时候。当 IP 指定为 127.0.0.1 时，没有这个问题（但指定为 localhost 也没法从外部访问了，没啥意义）。
